# Guidelines

## Product Rules

- 현재 프로젝트는 정적 `HTML + CSS + JS` 기반 단일 화면 앱 구조다.
- 브라우저에서 `index.html`을 직접 열어 실행하는 것을 기본 배포 방식으로 한다.
- `main.js`는 화면 렌더링, 상태 관리, 이벤트 처리, 모달, 토스트, 목록/상세, 검색, 필터, 페이징 로직을 모두 포함한다.
- `styles.css`는 모든 화면의 스타일을 포함하며 외부 스타일 프레임워크 의존성을 두지 않는다.
- 외부 CDN 의존성은 원칙적으로 두지 않으며, 필요한 리소스는 로컬 파일로 포함한다.

## Visual Rules

- 기존 B2B 관리자 UI 톤을 유지한다.
- 새로운 레이아웃이나 장식 요소를 추가하지 않는다.
- 목록/상세/모달 중심의 화면 구성을 유지한다.
- 기능과 디자인의 1:1 유지가 우선이다.

## Content Rules

- 문구는 기존 화면 기준을 유지한다.
- 버튼 이름, 상태 문구, 안내 문구를 임의로 변경하지 않는다.
- 안내 메시지는 사용자 행동을 바로 이해할 수 있어야 한다.

## Structure Rules

| 구분 | 기준 |
|---|---|
| `app/*` | 사용하지 않음 |
| `features/*` | 사용하지 않음 |
| `components/*` | 사용하지 않음 |
| `api/*` | 사용하지 않음 |
| `types/*` | 사용하지 않음 |
| `index.html` | 단일 진입점 |
| `main.js` | 앱 로직 전체 |
| `styles.css` | 전역 스타일 전체 |
| `fonts/*` | 로컬 폰트 리소스 |

## State Rules

| 상태 | UI 기준 |
|---|---|
| `loading` | 버튼 비활성화, 로딩 문구 표시 |
| `empty` | 결과 없음 안내 표시 |
| `error` | 에러 메시지와 재시도 동선 표시 |
| `success` | 토스트 또는 성공 상태 표시 |
| `disabled` | 권한 없음 또는 입력 미완료 상태 |

- 로딩과 빈 상태는 별도 텍스트로 구분한다.
- 에러 상태는 기존 입력값을 유지한다.
- 상태 전환은 즉시 반영한다.

## Permission Rules

| 메뉴 | MASTER | OPERATOR |
|---|---:|---:|
| 대시보드 | O | O |
| 콘텐츠 관리 | O | X |
| 캐시 답변 관리 | O | X |
| 지식 기반 조회 | O | O |
| 피드백 관리 | O | O |
| 계정/권한 관리 | O | X |

- 메뉴 접근은 권한 기준으로 제어한다.
- 비권한 메뉴는 표시하지 않거나 비활성화한다.
- 본인 계정은 권한 변경과 비활성화 대상에서 제외한다.

## Search / Pagination

- 검색은 각 화면 내부 범위에만 적용한다.
- 검색 결과는 필터 후 목록 기준으로 갱신한다.
- 페이지네이션은 1부터 시작하며 범위를 벗어나면 clamp 한다.
- 긴 목록 화면에서는 내부 스크롤을 우선 사용한다.

## Error Policy

| 상황 | 정책 |
|---|---|
| 조회 실패 | 에러 상태와 재시도 버튼 표시 |
| 검색 실패 | 입력값 유지 + 안내 문구 표시 |
| 중복 | 등록 전 차단 |
| 권한 없음 | 버튼 비활성화 또는 화면 진입 차단 |
| 빈 결과 | 빈 상태 안내 표시 |

## Message Policy

| 유형 | 위치 | 기준 |
|---|---|---|
| 성공 | 화면 상단 토스트 | 2.5~3초 후 자동 종료 |
| 실패 | 토스트 또는 모달 하단 | 원인과 다음 행동 안내 |
| 권한 없음 | 버튼 하단 또는 비활성 상태 | 가능한 경우에만 안내 |
| 복사 완료 | 토스트 | 즉시 피드백 |

## Notes

- 현재 구현은 정적 파일 기반이므로 Vite / React / TypeScript 의존을 전제로 작성하지 않는다.
- 기능 문서와 코드가 다르면 코드 기준을 우선한다.
- 불필요한 분기나 예외는 별도 문서로 분리한다.
